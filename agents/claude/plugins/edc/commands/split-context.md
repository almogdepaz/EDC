# Split Context

Read `.context/full-context.md` and split it into logical pieces. Do NOT discard content — reorganize it.

## context.md (repo root)

Brief architecture overview extracted from the full analysis. An agent loading this file should immediately understand:
- what the system is (2-3 sentences)
- what the modules are and where their deep context lives (table with links to `.context/*.md`)
- who the actors are
- what the key end-to-end flows are (one-liner each)
- what the global invariants are
- where the trust boundaries are
- where cross-module coupling creates cascade risk

Must begin with `<!-- generated by /edc:build-context -->`.

No function-level detail. No analysis. Just a map so agents know where to look.

## .context/{module}.md (per logical grouping)

Split the remaining analysis into files by logical grouping. Every section from full-context.md must end up in exactly one module file — nothing gets dropped.

Each file must begin with:
```
<!-- generated by /edc:build-context -->
# {Module Name} — Deep Context
```

Group by whatever boundaries make sense for the codebase (directory structure, domain concepts, architectural layers). Every module gets a file — even simple ones. If the full analysis has a section about it, that section goes into a file.

## .context/issues.md

Extract all problems, risks, bugs, fragility points, and correctness concerns found during the analysis into a dedicated issues file. Anything that describes something WRONG or RISKY — not just how things work — goes here.

Each issue must begin with:
```
<!-- generated by /edc:build-context -->
# Issues
```

List each issue with:
- brief description of the problem
- affected files/functions
- why it matters

Keep the same issue in the module file too (for context when reading that module), but collect them all in one place so they're actionable.

## .context/.meta.json

Generate a metadata file for incremental updates. Use `git rev-parse HEAD` to get the current commit.

```json
{
  "lastCommit": "<HEAD SHA>",
  "lastFullBuild": "<HEAD SHA>",
  "lastUpdate": "<ISO timestamp>",
  "modules": {
    "module-name": {
      "files": ["src/path/file1.ts", "src/path/file2.ts"],
      "contextFile": ".context/module-name.md"
    }
  }
}
```

The `files` array for each module should list all source files whose analysis ended up in that module's context file. This mapping is used by `/edc:update-context` to determine which modules need re-analysis when files change.
