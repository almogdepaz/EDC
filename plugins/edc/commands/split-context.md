# Split Context

Read `.context/full-context.md` and split it into:

## context.md (repo root)

Brief architecture overview. An agent loading this file should immediately understand:
- what the system is (2-3 sentences)
- what the modules are and where their deep context lives (table with links to `.context/*.md`)
- who the actors are
- what the key end-to-end flows are (one-liner each)
- what the global invariants are
- where the trust boundaries are
- where cross-module coupling creates cascade risk

Must begin with `<!-- generated by /edc:build-context -->`.

No function-level detail. No analysis. Just a map.

## .context/{module}.md (per logical grouping)

Split the full analysis into files by logical module grouping. Each file covers one coherent area of the codebase.

Each file must begin with:
```
<!-- generated by /edc:build-context -->
# {Module Name} — Deep Context
```

**Only include content that saves an agent time or prevents wrong assumptions.** If an agent can discover it with a single `Read`, `Grep`, or `Glob` — leave it out. The test: would a senior engineer who just joined the project say "I wish someone had told me this before I touched that code"?

Keep:
- invariants and implicit contracts not visible from reading one file
- cross-module coupling that would cause silent breakage
- non-obvious data flows spanning multiple modules
- fragility and gotchas (stateful logic, ordering constraints, TOCTOU gaps)
- why decisions were made when the rationale isn't in the code

Drop everything else. If a module has nothing non-obvious to say, don't create a file for it.
